## 前端代码提交规范

### git 库内容提交规范
禁止将 `node_modules` 和 `.map` 文件提交至 `git` 仓库，此类文件为开发期使用，会对项目造成破坏性的调整。

可以参考下列配置，对项目内 `.gitignore` 进行配置。
```
node_modules/
/maps/
/dist/
```

### commit 规范
前端代码提交 `commit` 格式由以下元素组成：
```
[单号] [代码提交类型]([修改模块])： [代码提交描述]
```

列子：
```
B00064355 fix(tl-input): 修复输入框组件显示问题
```

### 各元素详解

#### 单号
单号使用公司规范单号。
- 仅允许使用故事单、缺陷单作为提交代码的单号。
- 需为真实单号，且开发环节中需要存在提交代码的开发人员，和版本对应开发流。
- 故事单为 `S` 开头，缺陷单为 `B` 开头。

#### 代码提交类型
代码提交类型作用为标记此次提交的目的，由以下几种类型，请选择相应的类型填写至 `commit` 当中。
- feat：新功能（feature）
- fix：修复缺陷
- docs：文档编写（documentation）
- style：样式调整（不影响代码运行的变动）
- refactor：代码重构（即不是新增功能，也不是修改bug的代码变动）
- test：单元测试
- chore： 构建过程或辅助工具的变动

#### 修改模块
修改模块指的是当前调整的代码，在项目中的功能或维度。
列子：
我调整了 `yjpl-ui` 中的 `tl-input` 组件，则将 `tl-input` 作为该项的值。

#### 代码提交描述
对此次代码调整进行描述性总结。
